# AGENTS.md

Этот репозиторий предназначен для разработки алгоритмов **texturing**.

Цель этих правил — не формальность, а практическая инженерная дисциплина:
- чтобы проект можно было восстановить с нуля на новой машине;
- чтобы алгоритмы было проще отлаживать и профилировать;
- чтобы история задач, экспериментов и изменений не терялась;
- чтобы git-история оставалась чистой и понятной;
- чтобы не приходилось полагаться на случайное локальное состояние окружения.

Если какое-то правило конфликтует с удобством, по умолчанию нужно соблюдать правило, если только пользователь явно не сказал иначе.

---

## 1. Общие принципы работы

### Правила
- Предпочитай простые, практичные и инженерно разумные решения.
- Не добавляй лишнюю архитектуру без реальной необходимости.
- Не добавляй фичи, которые не были запрошены.
- Старайся менять код так, чтобы объём изменений оставался как можно меньше, если это не конфликтует с correctness и diagnosability.
- Особенно при большом рефакторинге предпочитай подход, где по возможности сохраняется работающий старый код, а новые изменения делаются поверх него небольшими и проверяемыми шагами.
- Если это помогает снизить риск и сделать diff понятнее, разбивай работу на несколько этапов: сначала refactoring без изменения логики, потом целевое функциональное изменение.
- Если что-то неоднозначно, выбирай самый простой разумный вариант и явно фиксируй его.
- Поддерживай репозиторий в состоянии, пригодном для продолжения работы позже.
- Саморефлексия по времени обязательна: каждый шаг, который может занять дольше примерно одной секунды, по возможности нужно явно замерять.
- В конце каждого логического шага работы по задаче, когда дан законченный результат на одну пользовательскую команду, нужно одной строкой писать краткое self-profiling summary:
  - какие основные типы деятельности заняли время;
  - сколько процентов общего времени они заняли;
  - в порядке убывания доли времени.
- Минимально имеет смысл различать хотя бы категории вроде:
  - ожидание сборки;
  - запуск тестов;
  - profiling;
  - чтение/поиск по коду;
  - собственно изменения кода;
  - размышление/анализ.
- Цель этого summary — не декоративная отчётность, а возможность заметить, если какой-то этап системно слишком дорогой, и вовремя принять инженерные меры.

### Зачем это нужно
Главный риск в таких проектах — не нехватка абстракций, а хаос, невоспроизводимость, потеря контекста и сложная отладка. Небольшие и поэтапные изменения снижают вероятность регрессий и упрощают уверенную проверку того, что после правки система всё ещё работает.

## 2. Репозиторий — источник истины

### Правила
- В репозитории должны храниться:
  - код;
  - конфиги-шаблоны;
  - скрипты запуска;
  - документация;
  - всё необходимое для восстановления рабочего состояния проекта.
- Важные знания о проекте не должны существовать только в голове разработчика или только в shell history.
- Если для запуска нужны ручные шаги, они должны быть задокументированы.
- Проект должен быть воспроизводим с нуля на новой машине.

### Зачем это нужно
Если локальная машина, build directory, venv или IDE-настройки пропадут, проект всё равно должен быть восстановим.

## 3. Язык кода, комментариев и документации

### Правила
- **Код пишется на английском**, включая:
  - имена сущностей;
  - названия файлов и API, если нет веской причины иначе.
- **Комментарии в коде пишутся на английском**.
- **Commit messages are written in English**.
- **Файлы документации пишутся на русском**, если нет особой причины использовать английский.
- Английские термины можно и нужно использовать там, где это технически естественно и уместно.

### Зачем это нужно
Код и комментарии на английском лучше сочетаются с инструментами, библиотеками и C++-экосистемой. Документация на русском удобнее для постановок задач, правил работы и инженерных пояснений в этом проекте. Английские commit messages лучше совместимы с будущей публичной историей, форками и upstream Pull Request workflow.

## 4. Секреты, токены и приватные локальные настройки

### Правила
- Никогда не коммить:
  - secrets;
  - tokens;
  - пароли;
  - API keys;
  - private credentials;
  - `.env` с реальными значениями;
  - machine-specific private config.
- Используй:
  - `env.example`;
  - шаблоны конфигов без секретов;
  - документацию с описанием необходимых переменных.

### Зачем это нужно
Секреты в git — это и проблема безопасности, и проблема сопровождения. Их трудно корректно удалить из истории.

## 5. Политика коммитов и push

### Правила
- Коммить только проверенные, логически завершённые изменения.
- Непроверенные эксперименты не коммить.
- Предпочитай маленькие атомарные коммиты.
- `commit message` должен быть коротким и понятным.
- `commit message` должен состоять из одной первой строки, которая в целом описывает изменение, и более подробного текста в следующих строках, который кратко перечисляет включённые изменения.
- После коммитов нужно делать **регулярные `git push`**.
- В целом допустимо делать `git push` **после каждого коммита**.
- Если идёт серия нескольких близких коммитов подряд, можно сначала сделать несколько коммитов, а потом одним блоком отправить их через `git push`.

### Зачем это нужно
Git-история должна показывать осмысленные шаги разработки, а удалённый репозиторий должен регулярно отражать текущее состояние работы, чтобы проект не жил только локально. Однострочный заголовок и последующее краткое пояснение делают историю коммитов одновременно обзорной и достаточно информативной.

## 6. Данные и большие артефакты

### Правила
- Не коммить датасеты и другие большие data artifacts по умолчанию.
- Не использовать git-репозиторий как хранилище сырых данных.
- При этом происхождение данных должно быть записано.
- Минимально допустимо хотя бы словесное описание вроде:
  - «предоставлено разработчиком».
- Лучше фиксировать данные в `docs/data_sources.md`, указывая, если известно:
  - имя;
  - источник;
  - назначение;
  - формат;
  - ограничения на использование.

### Зачем это нужно
Большие данные не должны захламлять git, но происхождение данных нельзя терять, иначе эксперименты перестают быть воспроизводимыми.

## 7. Prompt/task files

### Правила
- Prompt/task files всегда хранятся в формате **Markdown**.
- Они должны лежать под `prompts/`.
- Формат пути должен быть строго таким:

  `prompts/YYMM/DD/HHMMSS короткое описание.md`

- Пример:

  `prompts/2503/08/144454 создание изначального репозитория.md`

- Один prompt file соответствует одной логической задаче, но может охватывать несколько коммитов.
- Решение о том, является ли новое сообщение продолжением старой задачи или началом новой, принимает **Codex**.
- Prompt files должны хранить:
  - сообщения пользователя по задаче;
  - короткие логи действий;
  - короткие логи результатов.
- Prompt files не должны превращаться в полный шумный transcript.
- Если задача нетривиальная и требует подробного плана, profiling-процедуры или многошаговой архитектурной проработки, полный implementation-ready план нужно хранить в отдельном файле в `docs/`, а в `prompts/...` оставлять только краткий журнал и ссылку на соответствующий `docs/...`.
- После завершения такой задачи нужно обновить и подробный `docs/...`, и краткий `prompts/...`.

### Зачем это нужно
Git отвечает на вопрос «что изменилось в коде», а prompt files — на вопрос «какая задача решалась и как развивалась постановка».

## 8. Отчёты

### Правила
- По запросу нужно формировать лаконичные PDF-отчёты.
- Отчёты должны быть практичными, а не декоративными.
- Желательно держать размер отчёта в пределах примерно **до 50 MB**.
- В начале отчёта должен быть понятный заголовок:
  - о какой фиче;
  - о каком эксперименте;
  - или о каком наборе экспериментов идёт речь.
- После заголовка — короткое пояснение, если оно нужно.
- Далее допустимы:
  - картинки;
  - метрики;
  - before/after comparisons;
  - короткие подписи;
  - краткие выводы.
- Если изображения раздувают PDF, их нужно сжимать или уменьшать.

### Зачем это нужно
Отчёты нужны для быстрой и ясной фиксации результатов, а не для красивого оформления.

## 9. Профилирование и оптимизация

### Правила
- Нельзя оптимизировать вслепую.
- Сначала нужно получить данные:
  - timings по логам;
  - и/или profiler data.
- Потом принять решение.
- Потом менять код.
- Для CPU / основной части использовать **VTune**.
- Для CUDA использовать **NVIDIA Nsight**.
- При profiling API-level трассировки, например Vulkan API / driver API / runtime API, нужно смотреть не только на timeline и количество вызовов, но и на **суммарное время по каждому типу API-вызова**.
- Если какой-то отдельный API-вызов занимает заметную долю общего времени, это нужно явно отметить в выводах.
- В таком случае нужно отдельно подумать о практических вариантах:
  - переиспользовать соответствующий ресурс;
  - создавать/уничтожать его реже;
  - уменьшить частоту самого вызова;
  - убрать лишний churn вокруг этого API.
- Нельзя ограничиваться только выводом вроде «много submit-ов» или «есть пузыри на timeline», если profiler показывает, что существенная доля времени уходит в конкретные API-вызовы наподобие `vkCreateFence`, `vkDestroyFence`, `vkAllocate*`, `vkFree*` и т.п.
- При запуске GPU unit-tests по умолчанию сначала нужно запускать их на одной мощной дискретной видеокарте, используя `GPGPU_VISIBLE_DEVICES`, и только под конец прогонять на всех устройствах.
- Особенно это важно для profiling, сравнения скорости и быстрых итераций оптимизации: в первую очередь нужно ориентироваться на производительные дискретные GPU вроде NVIDIA или AMD.
- Если изменение нацелено именно на другой класс устройств, можно сразу запускать на них, но это должно быть осознанным исключением.
- Если запуск или profiling делается именно на дискретной GPU, это нужно делать **вне sandbox**, потому что sandbox может скрывать устройство, ломать инициализацию драйвера или давать нерепрезентативные результаты.
- При profiling производительности нужно использовать `RelWithDebInfo` / `ReleaseWithDebInfo`, а не `Debug`.
- При profiling нужно отключать сильно замедляющие вспомогательные механизмы, которые не являются предметом измерения. Например, для Vulkan profiling обычно нужно выключать `Vulkan Validation Layer`, если profiling не направлен именно на них.
- При profiling через NVIDIA Nsight удобно заранее ограничивать устройство через `GPGPU_VISIBLE_DEVICES`, чтобы использовалась только NVIDIA GPU.
- Если при profiling ожидается NVIDIA GPU, но она не видна, об этом нужно прямо сообщить, потому что без profiler data действовать вслепую неэффективно.
- Если нужный profiler не работает или не настроен, нельзя продолжать blind optimization — нужно явно сообщить о проблеме.
- Исследования и profiling начинать на **маленьких датасетах**.
- Перспективные выводы перепроверять на **более крупных датасетах**.
- Очень большие датасеты использовать уже для финальной проверки, когда это действительно нужно.

### Зачем это нужно
Без измерений оптимизация почти всегда либо бесполезна, либо направлена не в то место. Маленькие датасеты ускоряют цикл гипотеза → измерение → решение.

## 10. Runtime checks и `rassert`

### Правила
- В ключевом алгоритмическом коде нужно активно использовать `rassert`.
- `rassert` должен оставаться включённым и в **release builds**.
- Лучше использовать `rassert` чаще, чем реже.
- Допустимо потратить некоторую долю производительности на runtime-проверки.
- Не нужно заранее удалять полезные проверки ради экономии 5–10% скорости.
- Если profiling покажет, что конкретная проверка реально дорогая, только тогда можно решать, ослаблять её, отключать или убирать.

### Зачем это нужно
В алгоритмическом коде нарушение инвариантов часто проявляется далеко от места реальной ошибки. Частые и явные runtime-проверки делают сбои локальными и быстро диагностируемыми.

---

## 11. Контракт `rassert`

### Правила
Каноническая форма:

`rassert(predicate, unique_code)`

Где:
- `predicate` — условие, которое должно быть истинным;
- `unique_code` — стабильный числовой идентификатор конкретного места проверки.

При нарушении `predicate`:
- `rassert` должен бросать `runtime exception`;
- сообщение об ошибке должно содержать:
  - `unique_code`;
  - `file`;
  - `line`;
  - желательно `function`.

Для этого нужно использовать `std::source_location` из **C++20**.

### Зачем это нужно
Ошибка должна быстро локализоваться и однозначно находиться в логах.

---

## 12. Политика `unique_code`

### Правила
- `unique_code` должен быть **стабильным литералом**, а не runtime-значением, меняющимся от запуска к запуску.
- Схема генерации:
  - московское время в формате `YYYYMMDDhhmmss`;
  - затем номер строки в файле в виде 5 цифр.
- Пример:

  `2026030815340900239`

- После появления в коде `unique_code` не должен меняться без серьёзной причины.

### Зачем это нужно
Стабильные коды удобно искать в логах, сопоставлять между разными запусками и использовать для диагностики.

---

## 13. Где именно нужно использовать `rassert`

### Правила
`rassert` нужно щедро использовать в местах вроде:
- postconditions функций;
- algorithm invariants;
- проверок отсортированности;
- ограничений на размеры частей графа;
- согласованности связанных структур данных;
- предположений, пришедших из далёких мест кода;
- sanity checks после преобразований;
- важных preconditions, которые должны падать громко.

Примеры:
- Если функция разбивает граф на компоненты размером не больше 100, в конце нужно явно проверить, что каждая компонента имеет размер `<= 100`.
- Если код опирается на sorted array, это нужно явно проверить.
- Если алгоритм зависит от корректности размеров, индексов, связности или взаимного соответствия массивов, это нужно проверять.

### Зачем это нужно
Главный враг здесь — не лишние проверки, а тихое существование некорректного состояния.

---

## 14. Безопасный доступ к изображениям и пикселям

### Правила
- Доступ к пикселям должен быть **bounds-checked**.
- Выход за границы должен приводить к громкой ошибке, а не к тихому undefined behavior.
- По возможности ошибка должна указывать **на место вызова**, а не только на внутренности image-класса.
- Для этого желательно использовать `std::source_location`.
- Если есть доступ к пикселю через operator/function, безопасная проверка должна быть стандартным поведением по умолчанию, если только нет отдельно обоснованного unsafe-path.

### Зачем это нужно
Out-of-bounds доступ к изображениям — одна из самых частых и неприятных ошибок в коде, связанном с image/texturing/vision-style обработкой.

---

## 15. Зависимости и время сборки

### Правила
- Время компиляции имеет значение как инженерное ограничение.
- Если нет явной необходимости использовать `Debug`, нужно по умолчанию использовать `RelWithDebInfo`.
- По умолчанию компиляцию нужно запускать с использованием всех доступных на машине потоков CPU.
- Нельзя бездумно размазывать тяжёлые `header-only` зависимости по кодовой базе.
- Небольшие `header-only` библиотеки допустимы, если это оправдано.
- Если зависимость тяжёлая, её лучше прятать за:
  - маленький публичный `.h`;
  - `.cpp`-реализацию;
  - минимальный proxy API.
- Избегай лишних include в публичных заголовках.

### Зачем это нужно
Даже если runtime performance важнее всего, медленная сборка ухудшает скорость итераций и разработки.
`RelWithDebInfo` позволяет сохранять оптимизации для реалистичных замеров и поведения кода, но при этом оставляет отладочные символы для диагностики.
Использование всех доступных потоков CPU ускоряет цикл правка → сборка → проверка без изменения логики программы.

---

## 16. Ожидания к качеству алгоритмического кода

### Правила
- Предпочитай явные инварианты скрытым предположениям.
- Предпочитай диагностируемые ошибки молчаливому undefined behavior.
- Предпочитай простые и устойчивые инженерные решения хрупким clever-трюкам, если нет явных причин делать иначе.
- Любые performance-решения должны опираться на измерения.
- Если появляется нетривиальное предположение в алгоритме, подумай, не нужно ли его явно проверять через `rassert`.

### Зачем это нужно
Алгоритмический код быстро обрастает невидимыми предположениями. Если их не контролировать, они превращаются в дорогие баги.

---

## 17. Документация

### Правила
- Документация должна быть краткой, но полезной.
- По документации должно быть понятно:
  - что это за проект;
  - как подготовить окружение;
  - откуда берутся данные;
  - как хранятся задачи;
  - как делаются отчёты;
  - какие действуют profiling/runtime rules.
- Предпочитай документацию, которая помогает реально работать, а не создаёт шум.
- Для нетривиальных задач с несколькими этапами, profiling или архитектурными решениями подробный рабочий план и итоговые инженерные выводы нужно сохранять в отдельном `docs/...md` файле, чтобы их можно было проанализировать позже без восстановления контекста по коммитам или shell history.
- Если по задаче создаётся `prompts/...` файл, он должен кратко ссылаться на соответствующий подробный `docs/...` файл, а не дублировать его целиком.

### Зачем это нужно
Документация нужна не ради галочки, а чтобы проект можно было возобновить, проверить и развивать дальше.

---

## 18. Если чего-то не хватает или что-то сломано

### Правила
- Если не хватает инструмента, profiler, dataset или dependency, нужно прямо об этом сказать.
- Нельзя делать вид, что profiling или optimization были выполнены корректно, если нужный инструмент не работал.
- Нельзя молча обходить ключевые правила безопасности, воспроизводимости и диагностики.
- Если остаётся неопределённость, выбирай самый простой разумный вариант и явно его описывай.

### Зачем это нужно
Ложная уверенность хуже честно зафиксированного ограничения.

---

## 19. Чего делать не надо

### Правила
- Не коммить датасеты по умолчанию.
- Не коммить secrets.
- Не добавляй спекулятивную инфраструктуру «на вырост» без запроса.
- Не оптимизируй без измерений.
- Не убирай полезные runtime checks ради преждевременной микрооптимизации.
- Не превращай prompt files в полный transcript.
- Не рассчитывай на вечную сохранность локального состояния машины.

### Зачем это нужно
Это основные типовые способы испортить такой проект.

---

## 20. Приоритеты при принятии решений

Если несколько факторов конфликтуют, по умолчанию придерживайся такого порядка приоритетов:

1. correctness
2. diagnosability
3. reproducibility
4. safety of development workflow
5. clarity of history and documentation
6. iteration speed
7. performance tuning

### Зачем это нужно
Быстрый, но непрозрачный и невоспроизводимый алгоритмический проект — плохой проект. Производительность важна, но только после корректности, диагностики и воспроизводимости.
